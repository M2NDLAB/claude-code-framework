## IMP-036 design proposal (awaiting decision — NOT applied)

**D1 — Where it lives and what it is.** `.claude/framework-version`, at the ROOT of `.claude/`,
OUTSIDE `memory/`. Reasons: (a) the upgrade writes it at closing → if it lived in `memory/`
it would violate the "empty diff on `.claude/memory/`" invariant of Step 5; (b) it is none
of the three classes (METHOD/MEMORY/HYBRID): it is **GRAFT STATE** — a fourth species to be
declared explicitly in the procedure's taxonomy: written by the setup and by the
closing of the upgrade, NEVER reconciled 3-way, NEVER copied from the framework repo; (c) in the
framework repo it does NOT exist and is not created (the framework is not grafted: its version
is the git tags) — hence no collisions on copy. Verified: `.gitignore` does not
exclude it (under `.claude/` it ignores only `settings.local.json`) → it is committable; TREE.md
lists only directories → no template to touch.

**D2 — Format.** Flat `key: value` lines, ASCII, readable with cat/grep — no
JSON (it would invite jq: a useless dependency for 3 fields):
```
version: v0.5.1
commit: <sha of the tag's commit, read in the framework repo>
grafted: 2026-07-11
```
- `version` = SemVer tag of the grafted framework. It is THE field (the base of the 3-way).
- `commit` = `git rev-parse <tag>^{commit}` run in the framework repo at setup:
  an exact pin even if a tag were moved; it disambiguates a graft from an untagged state.
  Degradation without dependencies: if the framework arrived as an export without git history,
  `commit: n/d`.
- `grafted` = date of the ORIGINAL graft, never changes. The upgrade rewrites
  `version`+`commit` and leaves `grafted`: the history of the transitions lives in the git log of the
  file itself (dashboard, not journal — same philosophy as STATE.md).
- Alternative considered and discarded: one-line JSON — no benefit for the real consumption
  (human reading + grep), the cost of a parser.

**D3 — Who writes it at setup.** ONE instruction at Step 1 of `SETUP.md` (right after the
copy list): create the file with the values of the tag just copied. It covers greenfield AND
brownfield-CASE-B with no duplicated text (the brownfield section already declares "steps 1-5
apply here too"). CASE A (framework already grafted) does NOT write the pin at setup: it
receives it from the upgrade's retrofit (D4) — at most a one-line clause in CASE A that
says so. No checkbox in Step 2: the pin is not a `[TO BE DEFINED AT SETUP]` marker,
the source stays a single one.

**D4 — Upgrade: reading and retrofit.**
- Step 0: new preference **0. Read `.claude/framework-version`** → `version` is the
  baseline, end of the step. The current ask/estimate/degrade chain stays as a fallback
  for pre-pin grafts (the opening "the project does not record which version it was
  grafted from" becomes conditional).
- Step 6 (closing): **write/update the pin to `vY`**; if the project does not have it
  (pre-pin graft), CREATE it now — that is the automatic retrofit: from the first useful upgrade
  onwards the baseline is certain. `grafted` if unknown: `n/d (retrofit YYYY-MM-DD)`.
- The box "No automation, for now" (SETUP.md ~286-293) must be updated: today it
  declares the pin deferred TOO — that would become false; only the command/script
  remains deferred (IMP-037).

**D5 — Agnosticity and dependencies.** Content = only framework state
(version/sha/date): zero project/stack/environment names. Written by hand
(echo/heredoc), read with cat/grep: no new script, hook, tool or permission.
Considered and excluded (scope): a check on the pin in `/lint-memory` — the pin is not
memory; we will revisit it only if a real case justifies it.

**D6 — Files touched by the application (estimate, all docs).** `SETUP.md` in 4 places
(Step 1 copy; box "No automation"; Step 0; Step 6 + a possible line in
CASE A and in the taxonomy of the classes); `CHANGELOG.md` `[Unreleased]`; `LEARNINGS.md`
(IMP-036 → Applied). NOT touched: README/badge (constraint), commands, scripts, templates of
`memory/`, TREE.md. Heaviness: below threshold — 2-3 tasks with one commit each on this
same branch, plan block in this note (IMP-034).
