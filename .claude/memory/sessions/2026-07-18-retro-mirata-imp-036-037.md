## Proposta di design IMP-036 (in attesa di decisione — NON applicata)

**D1 — Dove vive e cosa è.** `.claude/framework-version`, alla RADICE di `.claude/`,
FUORI da `memory/`. Motivi: (a) l'upgrade lo scrive in chiusura → se vivesse in `memory/`
violerebbe l'invariante "diff vuoto su `.claude/memory/`" del Passo 5; (b) non è nessuna
delle tre classi (METODO/MEMORIA/IBRIDO): è **STATO DELL'INNESTO** — quarta specie da
dichiarare esplicitamente nella tassonomia della procedura: scritto dal setup e dalla
chiusura dell'upgrade, MAI riconciliato a 3 vie, MAI copiato dal repo-framework; (c) nel
repo-framework NON esiste e non si crea (il framework non è innestato: la sua versione
sono i tag git) — quindi niente collisioni alla copia. Verificato: `.gitignore` non lo
esclude (sotto `.claude/` ignora solo `settings.local.json`) → è committabile; TREE.md
elenca solo directory → nessun template da toccare.

**D2 — Formato.** Righe piatte `chiave: valore`, ASCII, leggibili con cat/grep — niente
JSON (inviterebbe jq: dipendenza inutile per 3 campi):
```
version: v0.5.1
commit: <sha del commit del tag, letto nel repo del framework>
grafted: 2026-07-11
```
- `version` = tag SemVer del framework innestato. È IL campo (la base del 3-way).
- `commit` = `git rev-parse <tag>^{commit}` eseguito nel repo del framework al setup:
  pin esatto anche se un tag venisse spostato; disambigua un innesto da stato non taggato.
  Degradazione senza dipendenze: se il framework è arrivato come export senza storia git,
  `commit: n/d`.
- `grafted` = data dell'innesto ORIGINALE, non cambia mai. L'upgrade riscrive
  `version`+`commit` e lascia `grafted`: lo storico delle transizioni sta nel git log del
  file stesso (cruscotto, non diario — stessa filosofia di STATE.md).
- Alternativa considerata e scartata: JSON one-line — nessun beneficio al consumo reale
  (lettura umana + grep), costo di un parser.

**D3 — Chi lo scrive al setup.** UNA istruzione al Passo 1 di `SETUP.md` (subito dopo la
lista di copia): crea il file coi valori del tag appena copiato. Copre greenfield E
brownfield-CASO-B senza testo duplicato (la sezione brownfield dichiara già "i passi 1-5
valgono anche qui"). CASO A (framework già innestato) NON scrive il pin al setup: lo
riceve dal retrofit dell'upgrade (D4) — al più una clausola di una riga nel CASO A che lo
dice. Niente checkbox nel Passo 2: il pin non è un marcatore `[DA DEFINIRE AL SETUP]`,
la fonte resta una sola.

**D4 — Upgrade: lettura e retrofit.**
- Passo 0: nuova preferenza **0. Leggi `.claude/framework-version`** → `version` è la
  baseline, fine del passo. La catena attuale chiedi/stima/degrada resta come fallback
  per gli innesti pre-pin (l'incipit "il progetto non registra da quale versione è stato
  innestato" diventa condizionale).
- Passo 6 (chiusura): **scrivi/aggiorna il pin a `vY`**; se il progetto non ce l'ha
  (innesto pre-pin), CREALO ora — è il retrofit automatico: dal primo upgrade utile in
  poi la baseline è certa. `grafted` se ignota: `n/d (retrofit YYYY-MM-DD)`.
- Il riquadro "Nessuna automazione, per ora" (SETUP.md ~286-293) va aggiornato: oggi
  dichiara rimandato ANCHE il pin — diventerebbe falso; resta rimandato solo il
  comando/script (IMP-037).

**D5 — Agnosticità e dipendenze.** Contenuto = solo stato del framework
(versione/sha/data): zero nomi di progetto/stack/ambiente. Scrittura a mano
(echo/heredoc), lettura cat/grep: nessun nuovo script, hook, tool o permesso.
Considerato ed escluso (scope): un controllo del pin in `/lint-memory` — il pin non è
memoria; se ne riparla solo se un caso reale lo giustifica.

**D6 — File toccati dall'applicazione (stima, tutto doc).** `SETUP.md` in 4 punti
(Passo 1 copia; riquadro "Nessuna automazione"; Passo 0; Passo 6 + eventuale riga nel
CASO A e nella tassonomia delle classi); `CHANGELOG.md` `[Unreleased]`; `LEARNINGS.md`
(IMP-036 → Applicate). NON toccati: README/badge (vincolo), comandi, script, template di
`memory/`, TREE.md. Onerosità: sotto soglia — 2-3 task con un commit ciascuno su questo
stesso branch, blocco-piano in questa nota (IMP-034).
